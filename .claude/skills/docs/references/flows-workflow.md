# Flows Workflow (--preview only)

## Opt-in gate

`--preview` present → proceed with this workflow.
`--preview` absent → **stop immediately, do nothing**. No auto-detection of domain complexity.
This is the only trigger — never infer intent from project size or flow count.

Anti-patterns (forbidden):
- Drawing any diagram without `--preview` being explicitly passed.
- Detecting "this project is complex enough" to decide whether to draw.

---

## Init mode

### Step 1: Domain detection

Derive domain list from the scout reports already in context (Phase 1 output):

1. Examine the top-level `src/` or equivalent directory for module/domain subdirectories.
2. Cross-reference with scout summaries: a directory with ≥2 service/handler/repository files counts
   as a domain.
3. Build a proposed list: `[<domain-name>, ...]`

If the structure is ambiguous (flat repo, monorepo with irregular layout, unclear grouping):
→ Ask via `AskUserQuestion` (header `"Domain List"`) — present the inferred list as a starting
  point, let the user confirm, remove, or add entries before proceeding.

If the structure is clear (≥80% confidence):
→ Proceed directly; note inferred list in the turn's silent-decisions block.

---

### Step 2: Generate per-domain flow files

For each confirmed domain, create `docs/flows/<domain>-flow.md`.

**File template:**

```
---
domain: <domain-name>
sources:
  - <relative/path/to/domain/dir>
  - <additional source paths if any>
commit: <git rev-parse HEAD output>
---

# <Domain Name> — Business Flow

<2-4 sentence plain-English summary: what this domain does, who triggers it, what it produces.>

## Flow

\`\`\`mermaid
flowchart TD
  ...
\`\`\`

<!-- Keep ≤ 12 nodes. Syntax per mermaidjs-v11 (reference only — do NOT invoke as generator). -->
```

**Relevance gates — additional diagrams:**

| Diagram type | Add ONLY IF |
|---|---|
| Sequence diagram | Flow crosses ≥ 2 distinct services or actors (e.g. API → worker → external service) |
| State diagram | A core entity in the domain has ≥ 3 distinct lifecycle states |
| ER snippet | Domain owns ≥ 3 directly related entities/models |

If a gate is not met, **omit that section entirely** — do not add a placeholder or comment.

All diagrams: inline ` ```mermaid ` blocks, ≤ 12 nodes each, written as text (not generated by a tool).

---

### Step 3: Commit hash

Before writing any file, capture the HEAD commit:

```
git rev-parse HEAD
```

Use the full hash as the `commit` field in each file's frontmatter. If the repo has no commits
(freshly initialised), use the string `"initial"` and note it in the turn.

---

## Update mode

### Step 1: Staleness check (per existing flow file)

For each file in `docs/flows/`:

1. Read its frontmatter to extract `commit` and `sources`.
2. Run:
   ```
   git diff --name-only <commit>..HEAD -- <sources>
   ```
3. Interpret output:
   - **Non-empty** → one or more source files changed since the diagram was drawn.
     Regenerate the entire file (summary + all diagrams) and update `commit` to HEAD.
   - **Empty** → no source changes. Skip this file — do not rewrite it.

### Step 2: New domains

Compare the current domain list (derived via Step 1 of init mode above) against existing
`docs/flows/` files. Any domain without a corresponding file → create it (same template as init).

### Step 3: Removed sources

If a flow file's `sources` paths no longer exist in the repo:
→ **Warn the user** in the turn output: which file, which missing paths.
→ **Do NOT auto-delete** the flow file — the user decides whether to remove or remap sources.

---

## Fallbacks (git not available or unreliable)

Trigger any of these conditions → fall back to full regeneration of all flow files:

| Condition | Detection |
|---|---|
| Not a git repo | `git rev-parse HEAD` exits non-zero |
| Shallow clone | `git rev-parse --is-shallow-repository` returns `true` or exits non-zero (old git) |
| `commit` field in frontmatter is `"initial"` or missing | Parse frontmatter |
| Unknown / corrupt commit hash | `git cat-file -t <commit>` exits non-zero |

Fallback behaviour:
1. Warn the user (one line, state which condition triggered).
2. Regenerate every flow file from scratch using current scout context.
3. Set `commit` to the current HEAD (or `"initial"` if no commits exist).

---

## Output location

All files write to `docs/flows/<domain>-flow.md` (kebab-case domain name).
The `docs/flows/` directory is created on first write if it does not exist.

The Phase 3 size check in both init and update workflows must include `docs/flows/*.md` in the
`wc -l` glob.

---

## Polished / standalone diagrams

If the user wants interactive or zoomable output after flows are generated, suggest `/preview --diagram`.
Do NOT build a parallel generation mechanism — this workflow owns only the inline markdown files.
